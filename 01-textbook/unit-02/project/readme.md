# Proyecto #2: Reacathon

El segundo proyecto es **construir una aplicación React** que consuma una **API pública**.

### Requerimientos técnicos

Tu aplicación debe:

- **Consumir una API publica** – esto podría ser cualquier cosa, pero debe tener sentido para tu proyecto.
- **Tener varios componentes** - Al menos uno clásico y uno funcional.
- **La aplicación puede tener un enrutador(_router_)** - con varias "páginas", esto depende de tu discreción y si tiene sentido para tu proyecto.
- **Incluir wireframes** - que diseñaste antes de crear la aplicación.
- **Estar en línea** y accessible al público

## Entregables necesarios

- Una **aplicación funcional**, hospedada en algún lugar de internet
- Un **enlace a tu aplicación** en la sección de URL de tu repositorio de Github
- Un **repositorio git alojado en GitHub**, con un enlace a tu proyecto alojado y _commits_ frecuentes que se remontan al inicio del proyecto.
- Un archivo **`readme.md`** con:
- Explicaciones de las **tecnologías** utilizadas

  - Un par de párrafos sobre el **enfoque general que tomaste**
  - **Instrucciones de instalación** para cualquier dependencia
  - Enlace a tus **wireframes** - bocetos de las principales vistas / interfaces en tu aplicación
  - Descripciones de **problemas sin resolver** o **obstáculos importantes** que tu equipo tuvo que superar

## Maneras sugeridas para comenzar

- **No dudes en escribir código desechable** para resolver problemas a corto plazo.
- **Lee los documentos para las tecnologías / librerias / API que utilices**.
- **Escribe código DRY**.
- **Se coherente con tu estilo de código.**
- **Haz commits tempranos y a menudo.** No tengas miedo de romper algo porque siempre puede retroceder en el tiempo a una versión anterior.
- **Manten las historias de usuario pequeñas y bien definidas**, y recuerda: las historias de usuarios se centran en lo que necesita un usuario, no en las tareas de desarrollo que deben realizarse.
- **Escribe código sobre el que otro desarrollador no tendría que preguntarte**. ¿Tienen sentido sus convenciones de nomenclatura? ¿Otro desarrollador podría ver su aplicación y comprender qué es todo?
- **Haga que todo esté bien formateado.** ¿Está indentado, de manera consistente? ¿Podemos encontrar el inicio y el final de cada div, llave, etc.?
- **Comenta tu código.** ¿Alguien entenderá lo que sucede en cada bloque o función? Incluso si es obvio, explicar el qué y el por qué significa que otra persona puede recogerlo y obtenerlo.
- **Escribe pseudocódigo antes de escribir el código real.** Es útil pensar en la lógica de algo.
